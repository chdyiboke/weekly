用户和豆包的对话，流程详细描述？

分为6 大核心阶段：
用户输入发起请求 → 前端构建请求 & 建立 SSE 连接 → 后端豆包服务处理推理 → 前端流式接收 & 实时渲染 → 对话结束收尾 → 异常 / 中断处理

SSE 连接：Server-Sent Events、请求头设置：
Accept: text/event-stream、Cache-Control: no-cache、Connection: keep-alive


核心：后端豆包服务处理推理 

1. 上下文持久化与加载：从缓存 / 数据库读取该session_id的历史对话，维护上下文窗口，截断超长历史保证 Token 合规。
2. 业务逻辑处理（分场景）
普通对话：直接将拼接好的 prompt 送入大模型推理；
RAG 场景：先向向量库检索相关知识库文档，注入 prompt 后再推理；
Agent 工具调用：判断是否需要调用工具（搜索、计算器等），并行调用工具获取结果后再生成回答。
3. 大模型流式推理
LLM 逐 Token 生成文本，不等待全部生成完毕，实时将增量内容输出。
4. SSE 事件封装与推送
后端将增量内容封装为标准 SSE 格式：data: {"content": "增量文本", "status": "generating"}\n\n，持续推送给前端。
5. 生成结束处理
模型推理完成后，推送结束标记（如data: [DONE]\n\n），关闭 SSE 连接，同时将完整对话持久化到数据库。


怎么理解 Token 超限？
你发给 AI 的所有文字，都会被拆成 Token 计算数量。

什么是「上下文窗口」？
大模型有最大记忆容量，这个容量就是「最大 Token 限制」，比如：
普通模型：8k Token（约 6000 汉字）
长文本模型：16k/32k/128k Token


什么是 Token 超限？
你和豆包的完整对话内容（三部分总和）的 Token 数量，超过了模型的最大上限，就叫 Token 超限。
包含 3 部分内容：
历史对话（你问的 + 豆包答的，所有记录）
当前你的问题
系统提示词 / 知识库内容

Token 超限会发生什么？
上下文丢失

Token 超限 用户好像无感知，是怎么做到的？
系统在后台自动化处理超限问题，完全屏蔽了 “报错” 和 “上下文断裂” 的感知，让对话看起来始终连贯流畅。

1. 滑动窗口截断
2. 智能摘要压缩
3. 关键信息保留

当前市面上比较流行的模型 token 长度？
主流模型已全面进入 32k~256k Token 时代，前沿模型突破100 万 Token，国内头部模型普遍达到128k~256k
